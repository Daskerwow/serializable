# json_forge (Serializable Engine 2026) 🚀

A modern, declarative, and type-safe data serialization and immutable modification library for Dart and Flutter. It provides robust JSON parsing, automatic serialization, and deep nesting modifications without requiring heavy boilerplate or background code generation (`build_runner`).

Built on the foundation of Clean Architecture and adhering strictly to **SOLID**, **DRY**, and **KISS** principles, `json_forge` eliminates the sluggish development compilation cycles and massive `.g.dart` file footprints common in large projects.

---

## 🚀 Key Advantages

- **Zero Code Generation:** Say goodbye to `build_runner`, constant terminal file watches, and bloated autogenerated output. Models are configured declaratively in standard code, boosting compilation speeds and streamlining refactoring operations.
- **Absolute Runtime Type Safety:** By mapping properties explicitly using a schema-first `FieldSet<M>` architecture, the Dart compiler performs structural analysis at development time. The problem of runtime Type Erasure inside collections is completely mitigated using single-instantiation type-erased parameter wrappers.
- **Production-Grade Total Parsers:** All internal core parsers function as total mathematical functions (e.g., `intOrZero`, `dateTimeOrEpoch`). They normalize mismatched data types gracefully at runtime instead of crashing the thread, ensuring higher uptime and resilience against unexpected downstream REST API layout adjustments.
- **Deep Declarative copyWith Modifiers:** Modifying complex nested immutable entity hierarchies is simplified using the ModelBinder syntax engine (`$`). Deeply nestled states can be cleanly rewritten using target paths via the `at(...)` layout without manual, error-prone waterfall mapping methods.
- **Detailed Debugging Contexts:** Instead of generic, obscure `_TypeError` exceptions, the serialization engine isolation wraps execution to produce detailed, context-rich `SerializationError` trees detailing exact object field trajectories, expected types, and the offending raw JSON values.

---

## 🛠️ Architecture Overview

The framework is decoupled into distinct, single-responsibility modules:

| Module                        | Description                                                                                              |
| :---------------------------- | :------------------------------------------------------------------------------------------------------- |
| **`types.dart`**              | Foundation type signatures for atomic combinators (`Parser<T>`), (`Serializer<T>`), and system mappings. |
| **`errors.dart`**             | Hierarchical immutable exceptions supplying detailed stringified diagnostic telemetry.                   |
| **`field.dart`**              | Property descriptors binding specific JSON mapping keys directly to Dart instance fields.                |
| **`parser.dart`**             | An analytical suite of sanitizing total primitive parsers, collection builders, and validation filters.  |
| **`extension.dart`**          | Syntactic sugar allowing developers to cleanly instantiate field mapping schemas from string literals.   |
| **`serializable_model.dart`** | Deep-path traversal tree mutation engines and global execution null guards.                              |
| **`model_type.dart`**         | The transport mechanism layer regulating state change vectors (`FieldPatch`).                            |

## 📖 Complete Usage Guide

### 1. Define Your Model

Declare complex, nested entity models that include enums, lists of primitive dates, maps with customized keys, and dependent relationships using a declarative schema approach:

```dart
import 'package:equatable/equatable.dart';
import 'package:json_forge/json_forge.dart';

enum DeviceStatus { active, maintenance, offline, unknown }
enum AccessLevel { read, write, execute }

// ══════════════════════════════════════════════════════════════════════════════
// Dependent Model: Sensor
// ══════════════════════════════════════════════════════════════════════════════
class Sensor extends Equatable with Serializable<Sensor> {
  const Sensor(this.uid, this.value, this.history);

  final String uid;
  final double value;
  final List<DateTime> history;

  static final $ = ModelType<Sensor, SensorFields>(Sensor.new, SensorFields());

  @override
  List<FieldOf<Sensor>> get fields => $.fields.all;

  static Sensor fromJson(Json json) => $.call(json);

  Sensor copyWith(Iterable<FieldPatch> Function(SensorFields $) updates) =>
      $.bind(this)(updates);

  @override
  List<Object?> get props => [uid, value, history];
}

final class SensorFields extends FieldSet<Sensor> {
  final uid = 'sensor_uid'.field<Sensor, String>((m) => m.uid);
  final value = 'last_value'.field<Sensor, double>((m) => m.value);
  final history = 'history_logs'.field<Sensor, List<DateTime>>(
    (m) => m.history,
    parser: listOf(dateTimeOrEpoch),
  );

  @override
  late final all = <FieldOf<Sensor>>[uid, value, history];
}

// ══════════════════════════════════════════════════════════════════════════════
// Root Model: Terminal
// ══════════════════════════════════════════════════════════════════════════════
class Terminal extends Equatable with Serializable<Terminal> {
  const Terminal(this.id, this.title, this.status, this.sensors, this.tokens);

  final int id;
  final String title;
  final DeviceStatus status;
  final List<Sensor> sensors;
  final Map<AccessLevel, String> tokens;

  static final $ = ModelType<Terminal, TerminalFields>(Terminal.new, TerminalFields());

  @override
  List<FieldOf<Terminal>> get fields => $.fields.all;

  static Terminal fromJson(Json json) => $.call(json);

  Terminal copyWith(Iterable<FieldPatch> Function(TerminalFields $) updates) =>
      $.bind(this)(updates);

  @override
  List<Object?> get props => [id, title, status, sensors, tokens];
}

final class TerminalFields extends FieldSet<Terminal> {
  final id = 't_id'.field<Terminal, int>((m) => m.id);
  final title = 'display_name'.field<Terminal, String>((m) => m.title);

  final status = 'device_status'.field<Terminal, DeviceStatus>(
    (m) => m.status,
    parser: enumOrDefault(DeviceStatus.values, DeviceStatus.unknown),
  );

  final sensors = 'attached_sensors'.field<Terminal, List<Sensor>>(
    (m) => m.sensors,
    parser: listOf(modelOf(Sensor.fromJson)),
  );

  final tokens = 'access_keys'.field<Terminal, Map<AccessLevel, String>>(
    (m) => m.tokens,
    parser: mapOf(enumOrDefault(AccessLevel.values), stringOrEmpty),
    serializer: (map) => {for (final e in map.entries) e.key.name: e.value},
  );

  @override
  late final all = <FieldOf<Terminal>>[id, title, status, sensors, tokens];
}
```

### 2. Use Your Model

```dart
void main() {
  // Simulating an incoming raw JSON network payload
  final Map<String, Object?> rawNetworkPayload = {
    't_id': 1024.0, // Double provided, will infer and normalize to an int
    'display_name': 'MAIN_GATE_TERMINAL',
    'device_status': 'active',
    'attached_sensors': [
      {
        'sensor_uid': 'SN-7000X',
        'last_value': 19.8,
        'history_logs': ['2026-06-21T18:30:00Z'],
      },
    ],
    'access_keys': {'read': 'token_secure_hash'},
  };

  // 1. Safe Deserialization
  final terminal = Terminal.fromJson(rawNetworkPayload);
  print('Parsed Terminal: ${terminal.title}, Status: ${terminal.status}');

  // 2. Safe, Type-Checked Deep Mutation via ModelBinder
  final updatedTerminal = terminal.copyWith(($) => [
    $.title.set('BACK_GATE_TERMINAL'),
    $.status.set(DeviceStatus.maintenance),
  ]);

  print('Mutated Title: ${updatedTerminal.title}');
  print('Mutated Status: ${updatedTerminal.status}');
}
```

## ⚡ Production Deployment Considerations

When integrating this library into high-throughput production server environments or real-time application runtime instances, review these performance and design patterns:

### 1. Centralized Error Aggregation Strategy

The framework relies heavily on structured exceptions subclassed from `SerializationError`. If an API contract breaks completely (e.g., a non-nullable core field receives an incompatible object signature), a detailed error log is generated.

> **Production Blueprint:** Always wrap remote or edge data ingest calls inside network layers using structured `try`/`catch` statements catching `SerializationError`. Forward these detailed stringified trace stacks directly to centralized tracking services (e.g., Sentry, Firebase Crashlytics) to observe API schema updates in real time.

### 2. Type Inference Engine (Smart Inference)

When omitting manual, fine-grained `Parser<T>` callbacks during property mappings, the underlying pipeline assigns fallback handlers (`_smartParse`).

If a legacy database returns an ambiguous type representation (such as floating-point `100.0` intended for a strict integer parameter), the framework safely converts it into a structural equivalent using internal zero-alloc total fallbacks (`intOrZero`, `doubleOrZero`), bypassing runtime crashes.

### 3. Avoiding Runtime Performance Bottlenecks during Type Erasure

When a `Field<M, R>` object wrapper is aggregated inside a generic structure list like `List<FieldOf<M>>`, the specific type parameter `R` undergoes type erasure in the Dart virtual machine. Invoking explicit parameter functions directly on these lists typically triggers unexpected `_TypeError` constraints.

- **The Architecture Solution:** The `Field` layout constructor generates an internally generic type-erased closure on structural instantiation: `_erased = (Object? v) => serializer(v as R)`. This achieves optimized $O(1)$ lookup and processing times during large entity conversions without relying on resource-intensive reflection mirrors (`dart:mirrors`).

### 4. Garbage Collection (GC) Optimization on Giant Trees

The execution mechanics behind deep `copyWith` rely internally on parsing a structural snapshot, iterating modifications via `SerializableHelpers.writeDeep`, and instantiating a fresh model clone.

For massive, heavily branched JSON structures exceeding 10+ levels of hierarchical depth, frequent continuous alterations may trigger memory allocations that put pressure on the Garbage Collector. For optimal system throughput, aim to design flatter, decoupled domain layouts.
